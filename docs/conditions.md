# Conditions

Condition helpers create Karabiner conditions.

```ts
import {
    bundleIdPattern,
    eventChanged,
    exceptEventChanged,
    exceptExpressionIsTrue,
    exceptFromDevice,
    exceptFromInputSource,
    exceptFromKeyboardType,
    exceptInApp,
    exceptVariableIs,
    exceptWithDeviceConnected,
    expressionIsTrue,
    fromDevice,
    fromInputSource,
    fromKeyboardType,
    inApp,
    variableIs,
    withDeviceConnected,
} from "karabiner-config-builder";
```

## Frontmost application conditions

Use `inApp()` to limit a binding to one or more frontmost apps.

```ts
inApp("com.apple.finder");
exceptInApp("com.apple.SystemSettings");
```

Multiple bundle identifiers can be provided:

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

Bundle IDs are treated as exact matches by default.

```ts
inApp("com.apple.finder");
```

Generates a pattern equivalent to:

```txt
^com\.apple\.finder$
```

Use `bundleIdPattern()` when you intentionally want to provide a raw regular expression pattern:

```ts
inApp(bundleIdPattern("^com\\.avid\\..*$"));
```

The same applies to `exceptInApp()`:

```ts
exceptInApp(bundleIdPattern("^com\\.apple\\..*$"));
```

## Variable conditions

Use `variableIs()` to require a Karabiner variable value.

```ts
variableIs("layer", 1);
```

Use `exceptVariableIs()` to exclude a variable value.

```ts
exceptVariableIs("layer", 1);
```

Variables are commonly used by `layer()` internally, but can also be used manually for custom modal behavior.

## Device conditions

Use `fromDevice()` to require a specific input device.

```ts
fromDevice({
    vendor_id: 1234,
    product_id: 5678,
});
```

Use `exceptFromDevice()` to exclude a device.

```ts
exceptFromDevice({
    vendor_id: 1234,
    product_id: 5678,
});
```

Multiple identifiers can be provided:

```ts
fromDevice(
    {
        vendor_id: 1234,
        product_id: 5678,
    },
    {
        vendor_id: 1111,
        product_id: 2222,
    },
);
```

## Connected device conditions

Use `withDeviceConnected()` to require that a device is connected.

```ts
withDeviceConnected({
    vendor_id: 1234,
    product_id: 5678,
});
```

Use `exceptWithDeviceConnected()` to exclude that connected-device state.

```ts
exceptWithDeviceConnected({
    vendor_id: 1234,
    product_id: 5678,
});
```

## Keyboard type conditions

Use `fromKeyboardType()` to require a keyboard type.

```ts
fromKeyboardType("ansi");
exceptFromKeyboardType("iso");
```

Multiple keyboard types can be provided:

```ts
fromKeyboardType("ansi", "jis");
```

## Input source conditions

Use `fromInputSource()` to require an input source.

```ts
fromInputSource({
    language: "en",
});
```

Use `exceptFromInputSource()` to exclude an input source.

```ts
exceptFromInputSource({
    language: "ja",
});
```

## Expression conditions

Use `expressionIsTrue()` for Karabiner expression conditions.

```ts
expressionIsTrue("system.use_fkeys_as_standard_function_keys");
```

Use `exceptExpressionIsTrue()` to exclude the same expression being true.

```ts
exceptExpressionIsTrue("system.use_fkeys_as_standard_function_keys");
```

## Event changed conditions

Use `eventChanged()` for `event_changed_if`.

```ts
eventChanged(true);
```

Use `exceptEventChanged()` for `event_changed_unless`.

```ts
exceptEventChanged(true);
```

## How Karabiner evaluates conditions

Multiple conditions on the same manipulator are evaluated as `AND`.

```ts
conditions: [inApp("com.avid.ProTools"), fromKeyboardType("ansi")];
```

That means:

```txt
in Pro Tools AND from ANSI keyboard
```

Multiple values inside one list-based condition behave like a match against any listed value.

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

That means:

```txt
in Finder OR in Ghostty
```

So this is useful:

```ts
conditions: [inApp("com.avid.ProTools"), fromDevice(moonlander)];
```

But this is usually impossible if emitted as two separate same-family conditions:

```ts
conditions: [inApp("com.apple.finder"), inApp("com.avid.ProTools")];
```

because it means:

```txt
in Finder AND in Pro Tools
```

KCB avoids this specific footgun when merging compatible rule-level and manipulator-level list conditions.

## Rule-level conditions

Karabiner stores conditions on individual manipulators, not on rules.

KCB supports rule-level conditions as a convenience by applying them to every manipulator generated by that rule.

```ts
rule(
    {
        description: "Pro Tools bindings",
        conditions: [inApp("com.avid.ProTools")],
    },
    bind("m", key("escape")),
    bind("p", key("spacebar")),
);
```

This is equivalent to putting the same condition on every manipulator in that rule.

Rule-level conditions also apply to manipulators generated by `layer()`.

```ts
rule(
    {
        description: "Finder navigation layer",
        conditions: [inApp("com.apple.finder")],
    },
    layer("caps_lock", {
        tapped: key("escape"),
        bindings: {
            h: key("left_arrow"),
            j: key("down_arrow"),
            k: key("up_arrow"),
            l: key("right_arrow"),
        },
    }),
);
```

## Merging with manipulator-level conditions

Rule-level conditions are merged with manipulator-level conditions.

```ts
rule(
    {
        description: "Pro Tools Moonlander bindings",
        conditions: [inApp("com.avid.ProTools")],
    },
    bind("m", key("escape"), {
        conditions: [fromDevice(moonlander)],
    }),
);
```

That behaves as:

```txt
in Pro Tools AND from Moonlander
```

Different condition families remain separate conditions.

## List condition merging

Compatible list-based conditions are merged and deduped.

```ts
rule(
    {
        description: "App bindings",
        conditions: [inApp("com.apple.finder")],
    },
    bind("m", key("escape"), {
        conditions: [inApp("com.mitchellh.ghostty")],
    }),
);
```

That becomes equivalent to:

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

This avoids accidentally generating impossible same-family `AND` conditions.

List merging applies to compatible list-style condition families such as:

```txt
frontmost application
device
connected device
keyboard type
input source
```

## Deduping

Duplicate values are deduped when compatible list conditions are merged.

```ts
rule(
    {
        description: "Finder bindings",
        conditions: [inApp("com.apple.finder")],
    },
    bind("m", key("escape"), {
        conditions: [inApp("com.apple.finder")],
    }),
);
```

This remains equivalent to:

```ts
inApp("com.apple.finder");
```

## Conflict detection

The builder rejects obvious impossible condition combinations.

### Conflicting variable values

```ts
rule(
    {
        description: "Invalid variable conditions",
        conditions: [variableIs("mode", 1)],
    },
    bind("m", key("escape"), {
        conditions: [variableIs("mode", 2)],
    }),
);
```

This fails because `mode` cannot be both `1` and `2`.

### Required and excluded variable value

```ts
rule(
    {
        description: "Invalid variable exclusion",
        conditions: [variableIs("mode", 1)],
    },
    bind("m", key("escape"), {
        conditions: [exceptVariableIs("mode", 1)],
    }),
);
```

This fails because `mode` is both required to be `1` and excluded from being `1`.

### Required and excluded app overlap

```ts
rule(
    {
        description: "Invalid app conditions",
        conditions: [inApp("com.apple.finder")],
    },
    bind("m", key("escape"), {
        conditions: [exceptInApp("com.apple.finder")],
    }),
);
```

This fails because Finder is both required and excluded.

### Conflicting event-changed values

```ts
conditions: [eventChanged(true), eventChanged(false)];
```

This fails because the event cannot be both changed and unchanged.

## Conservative validation

Conflict detection is intentionally conservative. It catches exact conflicts, but it does not try to fully analyze regular expressions or arbitrary expressions.

For example, this kind of overlap is not analyzed:

```ts
conditions: [
    inApp(bundleIdPattern("^com\\.avid\\..*$")),
    exceptInApp("com.avid.ProTools"),
];
```

A human can see that the patterns may overlap, but the builder does not attempt full regex reasoning.

When in doubt, prefer exact bundle IDs and simple condition combinations.
