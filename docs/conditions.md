# Conditions

Condition helpers create Karabiner conditions.

```ts
import {
    exceptInApp,
    fromDevice,
    inApp,
    variableIs,
} from "karabiner-config-builder";
```

## Frontmost application conditions

```ts
inApp("com.apple.finder");
exceptInApp("com.apple.SystemSettings");
```

Multiple bundle identifiers can be provided:

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

Bundle IDs are treated as exact matches by default.

```ts
inApp("com.apple.finder");
```

Generates a pattern equivalent to:

```txt
^com\.apple\.finder$
```

Use `bundleIdPattern()` when you intentionally want to provide a raw regular expression pattern:

```ts
inApp(bundleIdPattern("^com\\.avid\\..*$"));
```

## Variable conditions

```ts
variableIs("layer", 1);
exceptVariableIs("layer", 1);
```

## Device conditions

```ts
fromDevice({
    vendor_id: 1234,
    product_id: 5678,
});

exceptFromDevice({
    vendor_id: 1234,
    product_id: 5678,
});
```

Multiple identifiers can be provided:

```ts
fromDevice(
    {
        vendor_id: 1234,
        product_id: 5678,
    },
    {
        vendor_id: 1111,
        product_id: 2222,
    },
);
```

## Connected device conditions

```ts
withDeviceConnected({
    vendor_id: 1234,
    product_id: 5678,
});

exceptWithDeviceConnected({
    vendor_id: 1234,
    product_id: 5678,
});
```

## Keyboard type conditions

```ts
fromKeyboardType("ansi");
exceptFromKeyboardType("iso");
```

Multiple keyboard types can be provided:

```ts
fromKeyboardType("ansi", "jis");
```

## Input source conditions

```ts
fromInputSource({
    language: "en",
});

exceptFromInputSource({
    language: "ja",
});
```

## Expression conditions

```ts
expressionIsTrue("system.use_fkeys_as_standard_function_keys");
exceptExpressionIsTrue("system.use_fkeys_as_standard_function_keys");
```

## Event changed conditions

```ts
eventChanged(true);
exceptEventChanged(true);
```

## How Karabiner evaluates conditions

Multiple conditions on the same manipulator are evaluated as `AND`.

```ts
conditions: [inApp("com.avid.ProTools"), fromKeyboardType("ansi")];
```

That means:

```txt
in Pro Tools AND from ANSI keyboard
```

Multiple values inside one list-based condition behave like a match against any listed value.

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

That means:

```txt
in Finder OR in Ghostty
```

So this is useful:

```ts
conditions: [inApp("com.avid.ProTools"), fromDevice(moonlander)];
```

But this is usually impossible:

```ts
conditions: [inApp("com.apple.finder"), inApp("com.avid.ProTools")];
```

because it means:

```txt
in Finder AND in Pro Tools
```

## Rule-level conditions

Karabiner stores conditions on individual manipulators, not on rules.

The builder supports rule-level conditions as a convenience by applying them to every manipulator generated by that rule.

```ts
rule(
    {
        description: "Pro Tools bindings",
        conditions: [inApp("com.avid.ProTools")],
    },

    bind("m", key("escape")),
    bind("p", key("spacebar")),
);
```

## Merging with manipulator-level conditions

Rule-level conditions are merged with manipulator-level conditions.

```ts
rule(
    {
        description: "Pro Tools Moonlander bindings",
        conditions: [inApp("com.avid.ProTools")],
    },

    bind("m", key("escape"), {
        conditions: [fromDevice(moonlander)],
    }),
);
```

That behaves as:

```txt
in Pro Tools AND from Moonlander
```

Different condition families remain separate conditions.

## List condition merging

Compatible list-based conditions are merged and deduped.

```ts
rule(
    {
        description: "App bindings",
        conditions: [inApp("com.apple.finder")],
    },

    bind("m", key("escape"), {
        conditions: [inApp("com.mitchellh.ghostty")],
    }),
);
```

That becomes equivalent to:

```ts
inApp("com.apple.finder", "com.mitchellh.ghostty");
```

This avoids accidentally generating impossible same-family `AND` conditions.

## Conflict detection

The builder rejects obvious impossible condition combinations.

### Conflicting variable values

```ts
rule(
    {
        description: "Invalid variable conditions",
        conditions: [variableIs("mode", 1)],
    },

    bind("m", key("escape"), {
        conditions: [variableIs("mode", 2)],
    }),
);
```

This fails because `mode` cannot be both `1` and `2`.

### Required and excluded app overlap

```ts
rule(
    {
        description: "Invalid app conditions",
        conditions: [inApp("com.apple.finder")],
    },

    bind("m", key("escape"), {
        conditions: [exceptInApp("com.apple.finder")],
    }),
);
```

This fails because Finder is both required and excluded.

## Conservative validation

Conflict detection is intentionally conservative.

It catches exact conflicts, but it does not try to fully analyze regular expressions or arbitrary expressions.

For example, this kind of overlap is not analyzed:

```ts
conditions: [
    inApp(bundleIdPattern("^com\\.avid\\..*$")),
    exceptInApp("com.avid.ProTools"),
];
```

A human can see that the patterns may overlap, but the builder does not attempt full regex reasoning.
