# API Reference

This document covers the main builder API for authoring Karabiner configs in TypeScript.

For CLI usage, see [CLI docs](./cli.md).

## Import style

Installed configs should import from the package root:

```ts
import {
    bind,
    group,
    key,
    layer,
    profile,
    setup,
} from "karabiner-config-builder";
```

The package root exports the builder API, output helpers, condition helpers, integration helpers, and Karabiner types.

## `setup()`

`setup()` creates the full Karabiner config.

### Object style

```ts
setup({
    profiles: [
        profile({
            name: "Default",
            groups: [group("Basic bindings", bind("caps_lock", key("escape")))],
        }),
    ],
});
```

### Array style

```ts
setup([
    profile({
        name: "Default",
        groups: [group("Basic bindings", bind("caps_lock", key("escape")))],
    }),
]);
```

### Variadic style

```ts
setup(
    profile(
        {
            name: "Default",
            selected: true,
        },

        group("Basic bindings", bind("caps_lock", key("escape"))),
    ),
);
```

A config file should default-export the result:

```ts
export default setup(/* ... */);
```

Named exports are also supported:

```ts
export const config = setup(/* ... */);
```

## `profile()`

`profile()` creates a Karabiner profile.

Profiles can be written with a `groups` array:

```ts
profile({
    name: "Default",
    selected: true,
    groups: [group("Basic bindings", bind("caps_lock", key("escape")))],
});
```

Or with variadic groups:

```ts
profile(
    {
        name: "Default",
        selected: true,
    },

    group("Basic bindings", bind("caps_lock", key("escape"))),
    group("App bindings", bind("f18", key("spacebar"))),
);
```

If both `groups` and variadic groups are provided, the builder combines them with `groups` first.

```ts
profile(
    {
        name: "Default",
        groups: [group("Base bindings", bind("caps_lock", key("escape")))],
    },

    group("Extra bindings", bind("f18", key("spacebar"))),
);
```

Profiles may also include Karabiner profile settings such as `selected`, `virtual_hid_keyboard`, simple modifications, function keys, and other supported profile fields.

## `group()`

`group()` is the main KCB organization primitive.

A top-level group inside a profile compiles to a Karabiner complex modification rule.

```ts
group("Basic bindings", bind("caps_lock", key("escape")));
```

Groups can be written with an array of entries:

```ts
group("Basic bindings", [
    bind("caps_lock", key("escape")),
    bind("f18", key("spacebar")),
]);
```

Or with variadic entries:

```ts
group(
    "Basic bindings",

    bind("caps_lock", key("escape")),
    bind("f18", key("spacebar")),
);
```

Groups can also accept options:

```ts
group(
    {
        description: "Pro Tools bindings",
        conditions: [inApp("com.avid.ProTools")],
    },

    bind("m", key("escape")),
);
```

Group entries can be regular manipulators from `bind()`, generated layer definitions from `layer()`, or nested groups from `group()`.

Group-level conditions are builder sugar. Karabiner stores conditions on individual manipulators, so KCB applies group-level conditions to every manipulator generated by that group.

Nested groups inherit parent conditions:

```ts
group(
    {
        description: "Apple Magic Keyboard",
        conditions: [fromDevice(APPLE_MAGIC_KEYBOARD)],
    },

    group(
        {
            description: "Finder",
            conditions: [inApp("com.apple.finder")],
        },

        bind("return_or_enter", key("spacebar")),
    ),
);
```

That behaves as:

```txt
from Apple Magic Keyboard
AND in Finder
AND return_or_enter pressed
-> spacebar
```

See [conditions docs](./conditions.md) for merge and conflict behavior.

## `bind()`

`bind()` creates a Karabiner basic manipulator.

```ts
bind("caps_lock", key("escape"));
```

With options:

```ts
bind("m", key("mute"), {
    description: "Mute",
    modifiers: {
        mandatory: ["left_control"],
    },
});
```

With conditions:

```ts
bind("m", key("mute"), {
    conditions: [inApp("com.avid.ProTools")],
});
```

Tap/hold-style bindings are supported:

```ts
bind("caps_lock", key("left_control"), {
    tapped: key("escape"),
});
```

You can also run something after key up:

```ts
bind("f12", shell("echo tapped"), {
    held: shell("echo held"),
    finished: shell("echo finished"),
});
```

The main output is assigned to Karabiner’s `to`.

The option mappings are:

```txt
tapped   -> to_if_alone
held     -> to_if_held_down
finished -> to_after_key_up
```

## Key combo helpers

Combo helpers express modified input and output keys without manually writing Karabiner modifier objects.

Use them directly as `bind()` inputs:

```ts
import { bind, cmd, key, shift } from "karabiner-config-builder";

bind(cmd("spacebar"), key("f18"));
bind(cmd(shift("p")), key("print_screen"));
```

Use them inside `key()` for modified outputs:

```ts
bind("spacebar", key(cmd("spacebar")));
bind("p", key(cmd(shift("p"))));
```

They can be used on both sides of a binding:

```ts
bind(cmd("h"), key(opt("left_arrow")));
```

That means:

```txt
command + h -> option + left_arrow
```

### Generic modifiers

Generic helpers match either side of the modifier when used as an input:

```ts
cmd("a");
ctrl("a");
opt("a");
shift("a");
```

They produce the corresponding generic modifier when used as an output:

```ts
key(cmd("a"));
key(ctrl("a"));
key(opt("a"));
key(shift("a"));
```

Generic helpers map to Karabiner’s non-side-specific modifier names:

```txt
cmd   -> command
ctrl  -> control
opt   -> option
shift -> shift
```

### Side-specific modifiers

Use side-specific helpers when the physical modifier side matters:

```ts
leftCmd("a");
rightCmd("a");

leftCtrl("a");
rightCtrl("a");

leftOpt("a");
rightOpt("a");

leftShift("a");
rightShift("a");
```

They map to exact side-specific modifiers:

```txt
leftCmd    -> left_command
rightCmd   -> right_command

leftCtrl   -> left_control
rightCtrl  -> right_control

leftOpt    -> left_option
rightOpt   -> right_option

leftShift  -> left_shift
rightShift -> right_shift
```

For example:

```ts
bind(leftCmd("h"), key("left_arrow"));
bind("h", key(rightCmd("right_arrow")));
```

### Multiple modifiers

Nest helpers to build combinations:

```ts
cmd(shift("p"));
leftCmd(rightOpt("h"));
ctrl(opt(shift("spacebar")));
```

The same nesting works for outputs:

```ts
key(cmd(shift("p")));
key(leftCmd(rightOpt("h")));
```

Repeated modifiers are deduplicated:

```ts
cmd(cmd("h"));
```

is equivalent to:

```ts
cmd("h");
```

### Raw modifier options

Do not provide modifiers through both a combo helper and the corresponding raw modifier option.

For binding inputs, this is invalid:

```ts
bind(cmd("spacebar"), key("f18"), {
    modifiers: {
        mandatory: ["left_shift"],
    },
});
```

For key outputs, this is invalid:

```ts
key(cmd("spacebar"), {
    modifiers: ["left_shift"],
});
```

Use nested combo helpers instead:

```ts
bind(cmd(shift("spacebar")), key("f18"));
key(cmd(shift("spacebar")));
```

## `remap()`

`remap()` creates a Karabiner simple modification.

```ts
remap("caps_lock", "escape");
```

It can also accept object-style simple modification shapes when needed:

```ts
remap({ key_code: "caps_lock" }, { key_code: "escape" });
```

Use `remap()` for Karabiner simple modifications. Use `bind()` for complex modifications.

## Output helpers

Output helpers create Karabiner `to` events.

### `key()`

Outputs a key code:

```ts
key("escape");
```

Combo helpers add output modifiers:

```ts
key(cmd("tab"));
key(leftCmd("tab"));
key(cmd(shift("p")));
key(leftCmd(rightOpt("h")));
```

Generic modifier helpers produce generic modifiers:

```ts
key(cmd("tab"));
```

produces:

```txt
command + tab
```

Side-specific helpers produce exact modifiers:

```ts
key(leftCmd("tab"));
```

produces:

```txt
left command + tab
```

Raw modifier arrays remain available when working directly with Karabiner-style values:

```ts
key("tab", {
    modifiers: ["left_command"],
});
```

Prefer combo helpers for normal config authoring:

```ts
key(leftCmd("tab"));
```

Do not provide modifiers through both a combo helper and `options.modifiers`:

```ts
// Invalid
key(cmd("tab"), {
    modifiers: ["left_shift"],
});
```

Nest combo helpers instead:

```ts
key(cmd(leftShift("tab")));
```

Karabiner output options can still be provided alongside a combo:

```ts
key(cmd("tab"), {
    repeat: false,
});
```

Other output options include:

```ts
key("caps_lock", {
    holdDownMs: 100,
});

key("left_command", {
    lazy: true,
});
```

Passing a modifier key code outputs that physical modifier key:

```ts
key("left_command", {
    lazy: true,
});
```

That is distinct from wrapping another key with a modifier:

```ts
key(leftCmd("h"));
```

### `none()`

Outputs Karabiner’s virtual no-op key.

Use this when you want to swallow a key press or intentionally map something to nothing.

```ts
bind("caps_lock", none());
```

This emits:

```json
{
    "key_code": "vk_none"
}
```

Inside a layer:

```ts
layer("nav", {
    trigger: "caps_lock",
    tapped: key("escape"),

    bindings: [bind("semicolon", none())],
});
```

### `shell()`

Outputs a raw Karabiner `shell_command`.

```ts
shell("osascript -e 'display notification \"Hello\"'");
```

`shell()` is intentionally raw. Use it when you want to control the exact shell command string Karabiner receives.

### `script()`

Wraps a script path in a Karabiner-friendly shell command.

This is useful because Karabiner runs shell commands in a limited environment that may not include your normal interactive shell `PATH`.

```ts
script("~/.dotfiles/bin/convert-wav-to-mp3", {
    shell: "zsh",
    PATH: ["~/.dotfiles/bin"],
    requires: ["lame", "osascript"],
});
```

Available options:

```ts
script(path, {
    shell: "zsh",
    PATH: ["~/.dotfiles/bin"],
    requires: ["lame"],
    args: ["--verbose"],
});
```

- `shell`: `"zsh"` or `"bash"`. Defaults to `"zsh"`.
- `PATH`: additional path entries to prepend before the default command path.
- `requires`: commands that must exist before the script runs.
- `args`: arguments passed to the script.

The default command path includes:

```txt
/opt/homebrew/bin
/usr/local/bin
/usr/bin
/bin
/usr/sbin
/sbin
$PATH
```

Paths beginning with `~` are expanded through `$HOME`.

### `app()`

Opens or focuses an application by bundle identifier using Karabiner’s native `open_application` software function.

```ts
app("com.mitchellh.ghostty");
```

By default, `frontmost` is `true`.

```ts
app("com.mitchellh.ghostty", {
    frontmost: false,
});
```

To find a bundle identifier locally:

```sh
osascript -e 'id of app "Ghostty"'
```

### `url()`

Opens a URL.

```ts
url("https://karabiner-elements.pqrs.org/");
```

### `macro()`

macros multiple outputs into one output array.

```ts
macro(key("escape"), shell("echo done"));
```

`macro()` requires at least one output.

## `layer()`

`layer()` creates a modal key layer backed by Karabiner variables.

Layers use a semantic name plus an explicit trigger key:

```ts
layer("nav", {
    trigger: "caps_lock",
    tapped: key("escape"),

    bindings: [
        bind("h", key("left_arrow")),
        bind("j", key("down_arrow")),
        bind("k", key("up_arrow")),
        bind("l", key("right_arrow")),
    ],
});
```

Use it inside a group:

```ts
group(
    "Navigation layer",

    layer("nav", {
        trigger: "caps_lock",
        tapped: key("escape"),

        bindings: [
            bind("h", key("left_arrow")),
            bind("j", key("down_arrow")),
            bind("k", key("up_arrow")),
            bind("l", key("right_arrow")),
        ],
    }),
);
```

### Options

| Option       | Type                | Description                                                                              |
| ------------ | ------------------- | ---------------------------------------------------------------------------------------- |
| `trigger`    | `string`            | Required key code that activates the layer.                                              |
| `tapped`     | `To \| To[]`        | Optional output when the trigger is tapped alone.                                        |
| `bindings`   | `Manipulator[]`     | Bindings active while the layer is held. Usually created with `bind()`.                  |
| `layers`     | `LayerDefinition[]` | Child layers active while the parent layer is held.                                      |
| `holdDownMs` | `number`            | Hold duration applied to key outputs in `tapped`. Defaults to `100`. Use `0` to disable. |

Layer bindings use the same `bind()` helper as regular group bindings. Combo helpers therefore work for both layer inputs and outputs:

```ts
layer("nav", {
    trigger: "caps_lock",
    tapped: key("escape"),

    bindings: [
        bind("h", key("left_arrow")),
        bind(cmd("h"), key("left_arrow")),
        bind(shift("h"), key("home")),
        bind("l", key(leftCmd("right_arrow"))),
    ],
});
```

This distinction matters:

```ts
bind(cmd("h"), key("left_arrow"));
```

means:

```txt
layer + command + h -> left_arrow
```

while:

```ts
bind("h", key(leftCmd("left_arrow")));
```

means:

```txt
layer + h -> left command + left_arrow
```

### Nested layers

Nested layers go in the `layers` array:

```ts
group(
    "App launcher layer",

    layer("main", {
        trigger: "caps_lock",
        tapped: key("escape"),

        bindings: [bind("g", app("com.mitchellh.ghostty"))],

        layers: [
            layer("open", {
                trigger: "o",

                bindings: [
                    bind("c", app("com.openai.chat")),
                    bind("f", app("com.apple.finder")),
                ],
            }),
        ],
    }),
);
```

Layer names are converted into Karabiner variable names.

For example:

```txt
nav      -> layer_nav
open     -> layer_open
nav open -> layer_nav_open
```

## Conditions

Condition helpers scope bindings by app, device, keyboard type, input source, variables, expressions, and event-change state.

```ts
group(
    {
        description: "Finder only",
        conditions: [inApp("com.apple.finder")],
    },

    bind("f18", key("spacebar")),
);
```

For the full condition reference, see [conditions docs](./conditions.md).

## Integrations

Integration helpers create output actions for external tools.

### AeroSpace

```ts
aerospace("workspace 1");
```

This outputs a shell command that runs:

```sh
aerospace workspace 1
```

### SoundFlow

```ts
soundflow("my.soundflow.package.command");
```

This outputs a shell command that triggers a SoundFlow command.

## Types

Karabiner types are exported from the package root.

Example:

```ts
import type {
    KarabinerConfig,
    Manipulator,
    Rule,
} from "karabiner-config-builder";
```

`Rule` is the generated Karabiner rule type. User-authored configs should generally use `GroupDefinition` for KCB groups and `Manipulator` for reusable binding arrays.

```ts
import type { Manipulator } from "karabiner-config-builder";
import { bind, key } from "karabiner-config-builder";

export const mediaBindings: Manipulator[] = [bind("f18", key("play_or_pause"))];
```

Reusable groups can be exported too:

```ts
import type { GroupDefinition } from "karabiner-config-builder";
import { bind, group, key } from "karabiner-config-builder";

export const mediaGroup: GroupDefinition = group(
    "Media",

    bind("f18", key("play_or_pause")),
);
```
